## 🔐 为什么数据库无法控制资金？

### 1. 资金在智能合约中

```typescript
// 智能合约（链上，已部署，不可修改）
contract DepositVault {
    balance: 1000 TON;  // ← 资金在这里！
    
    // 提款必须满足合约逻辑
    receive("Withdraw") {
        // ✅ 验证签名
        // ✅ 验证金额
        // ✅ 验证用户余额
        // ✅ 多签授权（冷钱包）
        
        if (all_checks_pass) {
            send(money);  // 才能转账
        }
    }
}
```

**关键**：
- ✅ 合约代码部署后**不可修改**
- ✅ 提款必须经过**链上验证**
- ✅ 数据库**无法**绕过合约逻辑

### 2. 数据库只是"记事本"

```sql
-- PostgreSQL 数据库（只是记录）
CREATE TABLE user_balances (
    user_id VARCHAR(255),
    balance DECIMAL(20, 9),  -- ← 这只是显示用的！
    ...
);

-- 黑客修改数据库：
UPDATE user_balances SET balance = 999999 WHERE user_id = 'hacker';
-- ⚠️  数据库改了，但链上余额没变！
```

**当用户尝试提款时**：

```typescript
// 后端验证
const dbBalance = await db.query('SELECT balance FROM user_balances WHERE user_id = ?');
// dbBalance = 999999（被黑客改的）

// 但是！向合约发起提款时：
const tx = await contract.send('Withdraw', { amount: 999999 });

// 合约会验证：
// ❌ 检查：链上余额是否足够？
// ❌ 结果：实际只有 100 TON
// ❌ 拒绝交易！
```

**结论**：数据库的数字再大，链上余额不够就无法提款！

## 🎮 完整游戏流程（数据库的作用）

### 场景 1：充值

```
1. 用户发起充值
   ┌──────────────┐
   │  用户钱包    │ --100 TON--> DepositVault（链上）
   └──────────────┘
                          ⬇️
2. 监听链上事件
   Backend 监听区块链 ─────▶ 发现充值交易
                          ⬇️
3. 更新数据库（仅记录）
   INSERT INTO transactions (user, amount, type) VALUES ('0x...', 100, 'deposit');
   UPDATE user_balances SET balance = balance + 100;
                          ⬇️
4. 前端显示（读取数据库）
   "您的余额：100 TON" ◀── 从数据库读取（快速显示）
```

**如果黑客改了数据库**：
- ❌ 前端显示 999999 TON
- ✅ 但链上实际只有 100 TON
- ❌ 提款时合约会拒绝（余额不足）

### 场景 2：游戏下注

```
1. 用户下注
   Frontend ─────▶ Backend ─────▶ DiceGameV2（链上）
                                       ⬇️
2. 合约验证（链上）
   - ✅ 检查 DepositVault 余额
   - ✅ 验证签名
   - ✅ 扣除余额
                                       ⬇️
3. 生成随机数（链上）
   native_prepare_random() ──▶ 公平、不可预测
                                       ⬇️
4. 结果上链
   GameResult { user, bet, result, win } ──▶ 区块链记录
                                       ⬇️
5. 后端同步（仅记录）
   Backend 监听 ─────▶ INSERT INTO game_results (...);
```

**如果黑客改了数据库**：
- ❌ 改游戏结果为"赢"
- ✅ 但链上记录是"输"
- ❌ 提款时会验证链上记录，发现不一致，拒绝

### 场景 3：提款

```
1. 用户申请提款
   Frontend ─────▶ Backend ─────▶ 验证
                                    ⬇️
2. 后端验证（双重检查）
   const dbBalance = await db.query(...);  // 数据库余额
   const chainBalance = await contract.getBalance(user);  // 链上余额
   
   if (dbBalance !== chainBalance) {
       throw new Error('数据不一致！可能被篡改！');
   }
                                    ⬇️
3. 向合约发起提款
   contract.send('Withdraw', { user, amount });
                                    ⬇️
4. 合约验证（最终防线）
   - ✅ 检查链上余额（不看数据库）
   - ✅ 验证多签授权
   - ✅ 扣除余额
   - ✅ 转账到用户钱包
```

**黑客的攻击会失败**：

```
黑客：UPDATE user_balances SET balance = 999999;

用户申请提款：
Backend: dbBalance = 999999 ✅
         chainBalance = 100  ✅
         不一致！❌ 拒绝提款

即使黑客绕过后端验证：
Contract: 检查链上余额 = 100 TON
          申请提款 = 999999 TON
          余额不足！❌ 交易失败
```

## 🛡️ 多层防护机制

### 第 1 层：数据库只读（业务逻辑）
```sql
-- 限制数据库权限
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES TO ton_user;
-- 无 DROP、TRUNCATE 权限
```

### 第 2 层：应用层验证
```typescript
// 后端验证：数据库 vs 链上
if (dbBalance !== chainBalance) {
    throw new Error('数据被篡改！');
}
```

### 第 3 层：智能合约验证（最终防线）
```tact
// 合约检查链上余额（不相信任何外部输入）
require(self.balances.get(sender) >= amount, "Insufficient balance");
```

### 第 4 层：多签冷钱包
```tact
// 大额提款需要 2/3 签名
require(signatures >= 2, "Need at least 2 signatures");
```
